# Воспроизведение пошаговой генерации визуалов: задача 16 ЕГЭ (трапеция)

Этот каталог содержит самодостаточный набор артефактов для проверки
генерации пошаговых геометрических визуализаций на задаче 16 ЕГЭ
(трапеция с перпендикулярными диагоналями), используемой в качестве
сквозного примера в выпускной квалификационной работе.

Соответствует:

- **рис. 3.2** (`fig:ch3:trapezoid-steps`) в основном тексте раздела 3.8 диплома;
- **приложению В.4** «Полный пример работы конвейера на задаче № 16».

## Содержимое

```
task16_trapezoid_repro/
├── README.md                              ← этот файл
├── input/
│   └── task16.md                          ← формулировка задачи
├── expected/

│   ├── auto_task16.html                   ← HTML-выход полного прогона
│   ├── task16_trapezoid_4steps.html       ← HTML с 4 шагами (1:1 с рис. 3.2)

    └── run_online.sh                      ← воспроизведение через Together AI
```

## Два режима воспроизведения

### Офлайн (без обращения к языковой модели)

Подходит для быстрой проверки работоспособности движка рендеринга.
Координаты заданы вручную в `pipeline/geometry_task16_demo.py`,
шаги выполняются последовательно.

```bash
cd <корень репозитория>
pip install -r requirements.txt
bash examples/task16_trapezoid_repro/scripts/run_offline.sh
```

После запуска ожидается:

- `pipeline/output_geometry/task16_trapezoid.html` — методический HTML
  с семью шагами (более развёрнутый, чем 4 шага рис. 3.2);
- эталон 1:1 с рис. 3.2 — `expected/task16_trapezoid_4steps.html`;
  это статичный артефакт, отдельно перерисовывать его не нужно.

### Онлайн (полный прогон конвейера через Together AI)

Воспроизводит полную цепочку планировщик → конвертер → решатель
ограничений → SVG-рендерер. Используется языковая модель
`Llama-3.3-70B-Instruct-Turbo`, поэтому нужен действующий ключ Together AI.

```bash
cd <корень репозитория>
pip install -r requirements.txt
export TOGETHER_API_KEY="tgp_v1_..."
bash examples/task16_trapezoid_repro/scripts/run_online.sh
```

После запуска ожидается:

- `pipeline/output_geometry/auto_task16_plan.json` — план, возвращённый
  языковой моделью;
- `pipeline/output_geometry/auto_task16.html` — HTML с пошаговым
  визуализированным решением.

Сравните полученный план с эталоном:

```bash
diff -u examples/task16_trapezoid_repro/expected/auto_task16_plan.json \
        pipeline/output_geometry/auto_task16_plan.json
```

Точные координаты и формулировки шагов могут несущественно отличаться
между прогонами из-за стохастичности языковой модели. Структурно
ожидаются: четыре основных вершины `A`, `B`, `C`, `D`, вспомогательная
точка `F` на продолжении `AD`, отрезок `CF` параллельный `BD`, маркер
прямого угла в треугольнике `ACF` и высота `h` из вершины `C` на
основание `AF`.

## Что считается успешным воспроизведением

Сравнение проводится качественно по следующим пунктам:

1. **Структура плана.** В JSON присутствуют поля `points`, `steps`,
   `canvas`, и каждый шаг ссылается на корректные имена точек.
2. **Геометрическая корректность.** В итоговом HTML видна трапеция
   `ABCD` с правильным соотношением оснований `BC : AD = 4 : 9`,
   обе диагонали `AC` и `BD` проведены, в треугольнике `ACF`
   отображается прямой угол.
3. **Числовая корректность.** В подписях стороны `BC = 4`, `AD = 9`,
   `AC = 12`, `BD = 5` соответствуют условию задачи; при наличии шага
   с высотой ответ принимает значение `h = 60/13`.

В спорных случаях используется визуальное сравнение со скриншотом
`expected/task16_trapezoid_full.png`, который служит эталоном для
рис. 3.2 диплома и приложения В.4.

## Связь с тестами

Корректность вспомогательной логики, которую использует геометрическая
ветка, покрыта изолированными модульными тестами в репозитории:

- `pipeline/_smoke_geometry_variance.py` — тесты агрегаторов метрик
  серии повторных прогонов;
- `pipeline/_test_answer_validator.py` — тесты пост-валидатора
  визуализации ответа.

Эти тесты не требуют обращения к языковой модели и могут запускаться
из CI.
