# Выводы по разработке планировщика визуалов для конспектов ЕГЭ

## 1. Что было сделано

### 1.1 Разработан планировщик визуалов (visual planner)

Создан полноценный модуль `visual_planner.py`, который:

- **Парсит конспект** — разбивает markdown-текст на семантические секции по заголовкам (##/###)
- **Анализирует каждую секцию через LLM** — определяет необходимость визуала, его тип и параметры
- **Генерирует Scene JSON** — структурированное описание визуала, готовое для передачи в рендерер

### 1.2 Определена таксономия визуалов

Разработана система из **13 типов визуализаций**, покрывающая потребности конспектов ЕГЭ по математике:

| Категория | Типы |
|-----------|------|
| Графики | `function_graph`, `tangent_line_graph`, `coordinate_plane` |
| Аналитические схемы | `derivative_sign_chart`, `number_line` |
| Структурные | `step_by_step_diagram`, `flowchart`, `outcome_tree` |
| Информационные | `formula_card`, `table`, `comparison_chart` |
| Иллюстративные | `annotated_example`, `venn_diagram` |

Таксономия разработана на основе анализа 4 реальных конспектов (задания №4, №6, №8, №12) и рекомендаций из исследования ("Конспекты ЕГЭ Research.docx").

### 1.3 Протестировано на 4 конспектах x 3 модели

Проведено **12 экспериментов** (4 конспекта × 3 модели), в каждом из которых планировщик обработал 13-22 секций конспекта.

**Тестовые конспекты:**
- №4 — Классическая вероятность (22 секции)
- №6 — Уравнения (14 секций)
- №8 — Производная и графики (13 секций)
- №12 — Исследование функций (16 секций)

**Протестированные модели:**
- Qwen 2.5-7B-Instruct-Turbo (~7B параметров)
- Mistral Small 24B (~24B параметров)
- Google Gemma 3n-E4B (~4B эффективных параметров)

## 2. Ключевые результаты

### 2.1 Планировщик работает стабильно

- **100% JSON-парсинг** — все три модели стабильно генерируют валидный JSON
- **Контекстная адекватность** — типы визуалов соответствуют содержанию секций:
  - Для вероятности (task4) → `outcome_tree`, `formula_card`
  - Для уравнений (task6) → `formula_card`, `flowchart`, `annotated_example`
  - Для производных (task8) → `tangent_line_graph`, `derivative_sign_chart`
  - Для исследования функций (task12) → `function_graph`, `derivative_sign_chart`, `step_by_step_diagram`

### 2.2 Количественное сравнение моделей

| Метрика | Qwen 2.5-7B | Mistral Small 24B | Gemma 3n-E4B |
|---------|:-----------:|:-----------------:|:------------:|
| Визуалов из 63 секций | 28 (44%) | 46 (73%) | 38 (60%) |
| Среднее время на конспект | ~97s | ~60s | ~88s |
| Выходных токенов (всего) | 10,583 | 15,436 | 15,596 |
| Разнообразие типов | 7 типов | 10 типов | 11 типов |

### 2.3 Качественное сравнение моделей

**Qwen 2.5-7B-Instruct-Turbo:**
- Самый **селективный** — помечает только 44% секций как требующие визуала
- Хорошо фильтрует: вводные части, советы и мотивационные блоки корректно помечает как "визуал не нужен"
- Генерирует **лаконичные, но содержательные** params
- Единственная проблема: task6 (уравнения) — выдал только 1 визуал из 14 секций, что маловато

**Mistral Small 24B:**
- Самый **щедрый** — 73% секций получают визуалы
- Наибольшее **разнообразие** типов (10 из 13), включая `annotated_example` и `comparison_chart`
- Быстрее остальных (238s на все 4 конспекта)
- Проблема: может быть **избыточным** — некоторые визуалы не добавляют ценности

**Google Gemma 3n-E4B:**
- **Сбалансированный** вариант (60% покрытия)
- Хорошо работает с `flowchart` (4 штуки, больше других моделей)
- Самый медленный по времени
- Генерирует больше выходных токенов

### 2.4 Согласованность моделей

Интересное наблюдение: на ключевых секциях (формулы, примеры решений) модели **единодушны** в выборе типа визуала. Например:
- Секция «Пример: Найти наибольшее значение» — все три модели выбрали `function_graph` с приоритетом "high"
- Секция «Формула классической вероятности» — все выбрали визуал, хотя конкретный тип варьируется

Расхождения возникают на **пограничных секциях**: советы, списки рекомендаций, алгоритмы без конкретных примеров — одна модель считает, что визуал нужен, другая — нет.

## 3. Качество Scene JSON

Генерируемые Scene JSON **готовы к использованию рендерером**. Примеры:

**Хорошие результаты:**
- `outcome_tree` для задачи на вероятность содержит конкретные исходы и благоприятные исходы
- `formula_card` содержит LaTeX формулу и пояснения к компонентам
- `function_graph` содержит уравнение, диапазон x и highlight_points
- `derivative_sign_chart` содержит критические точки и интервалы со знаками

**Области для улучшения:**
- Некоторые `derivative_sign_chart` содержат абстрактные точки ("x1", "x2") вместо конкретных чисел — нужно извлекать из текста
- У `tangent_line_graph` иногда неверный slope (написано "negative" для x=2 на y=x², где наклон положительный)
- `outcome_tree` для простых случаев (1 монета) слишком тривиален — можно было бы не генерировать

## 4. Рекомендации

### 4.1 Выбор модели для продакшена

**Рекомендация: Qwen 2.5-7B-Instruct-Turbo** с возможным fallback на Mistral Small 24B.

Обоснование:
- Qwen — лучший баланс «достаточно визуалов, но не слишком много»
- Для конспектов, где Qwen выдаёт мало визуалов (как task6), можно подключать Mistral
- Стоимость: Qwen значительно экономичнее по токенам

**Альтернативная стратегия:** Запускать Mistral Small 24B и пост-фильтровать результаты по приоритету (оставлять только "high" и отдельные "medium").

### 4.2 Улучшение промпта

Текущий промпт работает хорошо, но можно улучшить:
1. Добавить явные примеры секций, где визуал **не нужен** (мотивационные блоки, советы)
2. Для task6 (уравнения) — добавить примеры визуалов для формульных преобразований
3. Добавить правило: "если в секции уже есть markdown-таблица, предложи визуальную альтернативу, а не дублируй"
4. Добавить правило минимальной длины секции для визуала

### 4.3 Следующие шаги в пайплайне

1. **Template Retrieval** — по `visual_type` из Scene JSON выбирать SVG/TikZ шаблон
2. **Renderer** — подставлять params в шаблон и генерировать SVG
3. **Персонализация** — уровень ученика влияет на подробность визуала:
   - Слабый уровень → больше аннотаций, подписей, пошаговых элементов
   - Продвинутый → более компактные схемы, меньше текста
4. **Верификация** — round-trip OCR (Mathpix/pix2tex) для проверки корректности

### 4.4 Оценка стоимости

При средних ~1,500 входных + ~170 выходных токенов на секцию и ~15 секциях на конспект:
- На один конспект: ~22,500 input + ~2,550 output токенов
- Стоимость через Together AI (Qwen 2.5-7B): порядка $0.003-0.005 за конспект
- При генерации 1000 конспектов/день: ~$3-5/день — крайне экономично

## 5. Итог

Разработанный планировщик визуалов **решает поставленную задачу**: автоматически анализирует текст конспекта ЕГЭ, определяет где нужны визуализации, выбирает подходящий тип из таксономии и генерирует структурированное описание (Scene JSON), пригодное для автоматического рендеринга.

Небольшая LLM (Qwen 2.5-7B) справляется с задачей на уровне, достаточном для продакшена, при минимальных затратах на инференс. Модели размером 24B (Mistral Small) дают бо́льшее покрытие и разнообразие, но требуют пост-фильтрации.

Система готова к интеграции с рендерером визуалов — следующим этапом пайплайна.
