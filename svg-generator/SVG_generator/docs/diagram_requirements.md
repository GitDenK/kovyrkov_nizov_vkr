# Требования к диаграммам (`scene_type=diagram`)

## 1. Цель

Определить единые требования к учебным диаграммам, чтобы результат был:
- понятным для ученика;
- технически валидным (SVG);
- воспроизводимым при повторном рендере;
- согласованным с текущей архитектурой сервиса.

Документ покрывает только диаграммы (`diagram`), не графики функций и не геометрические чертежи.

## 2. Поддерживаемые элементы (MVP)

Обязательная поддержка:
- контейнер с текстом (`Box`);
- стрелка между элементами (`Arrow`);
- подпись к стрелке (`Arrow.label`);
- произвольный текст без контейнера, с возможностью рендеринга формул (`Text`);
- заголовок диаграммы (`Title`);

Уточнение по стрелкам:
- должны поддерживаться минимум два маршрута: прямая и с одним углом;
- стрелка должна иметь наконечник;
- подпись стрелки должна быть читаемой и не сливаться с линиями/рамками.

## 4. Функциональные требования

1. Должен использоваться единый способ рендера математических выражений для всех текстовых элементов диаграммы.
2. Математические выражения должны корректно отображаться в рамках выбранного движка
3. Сервис должен автоматически вычислять размер SVG после раскладки элементов. Соотношение тсорон svg должно стремиться к 1:1
4. Сервис должен добавлять внешний отступ вокруг контента.
5. Сервис не должен оставлять чрезмерно большие пустые поля.
6. При повторном рендере одной и той же сцены должен получаться одинаковый SVG.

## 5. Визуальные и layout-требования

1. Элементы (контейнеры, заголовки, отдельные текстовые блоки, формульные блоки) не должны пересекаться.
2. Стрелки могут пересекаться только в исключениях; целевое поведение — минимизировать пересечения.
3. Текст внутри контейнера не должен выходить за его границы.
4. Подпись стрелки не должна перекрывать текст контейнеров.
5. Диаграмма должна иметь логичную структуру: сверху-вниз или слева-направо, с небольшими контролируемыми ветвлениями.
6. Заголовок должен быть визуально отделен от основного тела диаграммы.

## 6. Требования к тексту и формулам

1. Шрифт, размер и цвет текста должны быть согласованы в пределах одной диаграммы.
2. Многострочный режим обязателен для текста внутри контейнеров (`Box`).
3. Многострочный режим обязателен для подписей стрелок (`Arrow.label`).
4. Многострочный режим обязателен для свободного текста (`Text`) и заголовка (`Title`) при превышении доступной ширины.
5. Для формульных блоков (`FormulaBlock`) при нехватке ширины допускается перенос на несколько строк, если это поддерживает выбранный движок рендера.
6. Если формула не поддерживается движком, система должна вернуть предупреждение в `warnings`.
7. В диаграмме не должно быть пустых или полностью невидимых ключевых текстовых элементов.

## 7. Технические требования к SVG

1. Результат должен быть валидным XML/SVG.
2. В результирующем SVG не должно быть `NaN` и нечисловых координат.
3. Контент должен быть в пределах canvas с учетом допустимого технического запаса.
4. Обрезка содержимого по границе SVG должна быть включена (`overflow="hidden"` или эквивалент).
5. Должна сохраняться совместимость с встраиванием в веб-страницы без внешних зависимостей.

## 8. Ограничения MVP

1. Авто-layout допускается простой (без сложного графового оптимизатора), но обязан обеспечивать читаемость и отсутствие критичных наложений.
2. Интерактивность и анимации в требования диаграмм MVP не входят.

## 9. Критерии приемки диаграммы

Диаграмма считается принятой, если одновременно выполняются условия:
- SVG валиден и рендерится в браузере;
- все ключевые элементы из сцены присутствуют;
- текст читаем, контейнеры и подписи не имеют критичных наложений;
- формулы отображаются корректно либо отданы с явным предупреждением и fallback;
- итоговый размер изображения адекватен контенту (нет обрезаний и больших пустых полей);
- повторный рендер сцены воспроизводим.
