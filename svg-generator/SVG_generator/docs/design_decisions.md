# Отклонения от первоначальных требований

Документ фиксирует решения, принятые в ходе планирования, которые отличаются от того, что описано в `system_design.md` и `product_requirements.md`.

---

## 1. Normalizer удалён как отдельный модуль

**Что было в system_design.md:**
Отдельный шаг `Normalizer` (`app/normalizer/normalizer.py`) между Validator и Solver. Отдельная сущность `NormalizedScene`.

**Что принято:**
Логика нормализации (раскрытие shorthand-конструкций, простановка defaults, resolve refs) перенесена в `app/validation/validator.py`. Шаг один: `validate_and_normalize`.

**Обоснование:**
Для трёх поддерживаемых типов сцен объём нормализации небольшой. Отдельный модуль увеличивает количество файлов и шагов pipeline без реальной пользы на уровне MVP.

---

## 2. Классификатор `scene_type` не реализуется

**Что было в system_design.md:**
Компонент `classifier` или "lightweight route selection" внутри сервиса определяет тип сцены из текста запроса.

**Что принято:**
`scene_type` является обязательным полем `InputRequest` и передаётся вызывающей системой (основным сервисом платформы). SVG-генератор не занимается классификацией.

**Обоснование:**
Классификация типа визуализации — ответственность вызывающей системы, которая знает контекст задачи или конспекта. Встроенный классификатор усложняет сервис и добавляет точку ошибки без необходимости.

---

## 3. Хранение артефактов — минимальный набор

**Что было в system_design.md:**
Для каждого запроса рекомендовалось сохранять: `input`, `scene_type`, `raw_llm_output`, `validated_scene`, `normalized_scene`, `solved_scene`, `final_svg`, `warnings`, `renderer_version`.

**Что принято:**
Хранится только минимальный набор:
```
input, scene_json (validated), svg, warnings
```

**Обоснование:**
Промежуточные артефакты (`raw_llm_output`, `solved_scene`) полезны для глубокого дебага, но увеличивают объём хранения и усложняют storage-модуль. На этапе MVP достаточно финального `scene_json` и `svg`. При необходимости расширить — добавить `raw_llm_output` в Milestone 4 (quality loop).

---

## 4. Solver использует sympy

**Что обсуждалось:**
Два варианта: чистый `math`/`numpy` (быстрее, без зависимостей) или `sympy` (надёжнее, покрывает сложные пересечения).

**Что принято:**
`sympy` — единственный выбор для всего geometry solver.

**Обоснование:**
Школьные геометрические построения (высота, биссектриса, пересечение прямых) корректнее выразить через символьные вычисления. `sympy` избавляет от ручных формул и снижает вероятность ошибок в граничных случаях (вертикальные прямые, совпадающие точки и т.д.).

## 5. Layout Engine встроен в renderer

**Что было в system_design.md:**
Отдельный компонент Layout Engine (`app/layout/layout.py`). Отдельный шаг pipeline между Solver и Renderer: bounding box, scaling, margins, world-to-screen transform, размещение подписей, авторазмер маркеров.

**Что принято:**
Bounding box и world-to-screen transform реализуются как внутренние функции `renderer.py`. Отдельного файла и шага pipeline нет.

**Обоснование:**
Для трёх типов сцен MVP объём layout-логики — 20–30 строк. Отдельный модуль создаёт дополнительный слой абстракции без реальной пользы. Если в будущем layout разрастётся, выделить его несложно.

---

## 6. Объекты `Line` и `Ray` убраны из MVP

**Что было в system_design.md:**
В списке поддерживаемых геометрических объектов: `Line` (бесконечная прямая) и `Ray` (луч) со своими схемами и рендером.

**Что принято:**
`Line` и `Ray` из MVP исключены. Если условие задачи подразумевает прямую или луч, LLM описывает длинный `Segment` с точками за пределами видимой области. Renderer обрезает по viewBox.

**Обоснование:**
В задачах ЕГЭ по планиметрии бесконечные прямые и лучи встречаются редко. Эмуляция через `Segment` визуально неотличима в большинстве учебных сценариев. Убирает два типа объектов из схем и рендера.

## 7. Рендер формул через matplotlib mathtext

**Что не было зафиксировано в system_design.md:**
Способ рендера `FormulaBlock` в SVG не был определён.

**Что принято:**
Формулы рендерятся через `matplotlib.mathtext` → SVG `<path>` элементы. `matplotlib` добавляется в зависимости проекта.

**Обоснование:**
- Не требует внешней установки LaTeX
- Чистое Python-решение, работает на сервере
- Поддерживает дроби, степени, индексы, греческие символы — достаточно для задач ЕГЭ
- Альтернатива `<foreignObject>` не работает в standalone SVG, Unicode-текст не покрывает нужный набор символов

## 8. Валидация SVG-строки в оркестраторе

**Контекст:**
Renderer генерирует SVG через конкатенацию строк. При добавлении новых типов объектов или спецсимволов в тексте (например, `<`, `&` в подписи точки) может получиться невалидный XML — браузер его не отобразит, но исключения в pipeline не будет.

**Что принято:**
После `render()` оркестратор проверяет SVG-строку через `xml.etree.ElementTree.fromstring()`. Это стандартная библиотека Python, дополнительных зависимостей нет.

Поведение при ошибке: возвращается `RenderResult(svg=None, warnings=[...])` с описанием ошибки парсинга. HTTP 500 не бросается.

```python
try:
    ET.fromstring(svg)
except ET.ParseError as e:
    return RenderResult(svg=None, warnings=warnings + [f"SVG XML invalid: {e}"])
```

**Где валидация:** оркестратор (`orchestrator.py`), не renderer. Renderer отвечает за генерацию, оркестратор — за контроль качества результата.

**Почему не в renderer:**
Renderer не знает о `RenderResult` и не должен принимать решение о том, что вернуть клиенту. Это ответственность оркестратора.

**Почему `svg=None`, а не сломанный SVG:**
Невалидный XML браузер либо не отобразит совсем, либо отобразит частично и непредсказуемо. Лучше явно сообщить об ошибке, чем молча отдать нерабочий результат.

## Что осталось без изменений

- Три типа сцен (`function_plot`, `geometry`, `diagram`) — все три в MVP
- Pipeline: validate → solve → render (порядок сохранён)
- Принцип semantic-first: LLM описывает смысл, не координаты
- Golden-тесты как основной инструмент проверки ядра до подключения LLM
- Retry-политика для LLM (до 2 повторов при schema-ошибках)
- Structured output через `response_format`
- Orchestrator — отдельный модуль
