# Milestone 0 — Scene DSL и контракт данных

**Статус:** выполнен  
**Версия:** 0.1.0

---

## Цель

Зафиксировать единый контракт данных — Scene DSL — вокруг которого строится вся остальная разработка.

Без этого контракта нельзя:
- писать валидатор (не известно, что проверять);
- писать renderer (не известно, что рендерить);
- строить промпты для LLM (не известно, что генерировать);
- писать тесты (не известно, что считать корректным результатом).

---

## Реализованные артефакты

### 1. `app/schemas/common.py` — базовые типы

Содержит три вспомогательных модели, которые переиспользуются по всему проекту.

**`Canvas`**  
Описывает параметры холста: размер SVG в пикселях и диапазон мировых координат (`x_min`, `x_max`, `y_min`, `y_max`). Нужен, чтобы renderer мог вычислить world-to-screen transform — перевод математических координат в пиксели. Дефолты: 400×400 пикселей, диапазон ±5 по обеим осям.

**`Style`**  
Глобальный стиль сцены: тема (light/dark), цвет обводки, цвет заливки, размер и семейство шрифта. Позволяет задать единый внешний вид всей сцены без повторения параметров в каждом объекте.

**`ObjectStyle`**  
Локальное переопределение стиля для отдельного объекта. Если нужно выделить высоту треугольника красным пунктиром — задаётся здесь, не в глобальном Style. Поля: `stroke`, `fill`, `stroke_width`, `dash` (`solid` / `dashed` / `dotted`).

### 2. `app/schemas/objects.py` — объекты сцены

Все типы объектов реализованы как Pydantic-модели с полем `type` (literal string). Это позволяет использовать discriminated union — Pydantic сам определяет нужный класс при десериализации JSON.

**`Point`**  
Точка в мировых координатах (`x`, `y`). Базовый примитив — на него ссылаются почти все остальные объекты. Поле `label` позволяет подписать точку без создания отдельного `Label`-объекта (удобно для LLM).

**`Segment`**  
Отрезок между двумя точками. Ссылается на `id` объектов `Point` через поля `from_point` и `to_point`. Является заменой `Line` и `Ray`, которые исключены из MVP — длинный `Segment` с точками за пределами canvas визуально неотличим от прямой или луча, а renderer обрезает по viewBox.

**`Circle`**  
Окружность с центром в точке (`center` — `id` Point) и радиусом в мировых единицах. Используется в задачах на вписанные/описанные окружности.

**`Triangle`**  
Треугольник, заданный тремя вершинами (`vertices` — список из ровно трёх `id` Point). Отдельный тип (а не три отрезка) нужен для того, чтобы constraints (медиана, высота, биссектриса) могли ссылаться на треугольник как на единую сущность — solver знает, что это треугольник, и вычисляет его свойства соответствующим образом.

**`FunctionCurve`**  
График функции одной переменной. `expression` — Python-строка от `x` (например, `"x**2 - 2*x + 1"`). Solver сэмплирует функцию по заданному диапазону и передаёт точки в renderer. Поля `x_min`/`x_max` переопределяют диапазон холста для конкретной кривой.

**`Label`**  
Текстовая подпись, привязанная к точке (`anchor` — `id` Point). Смещение задаётся в мировых единицах через `dx`/`dy`. Отдельный тип нужен для аннотаций, которые требуют точного позиционирования или особого стиля (например, пояснение к формуле рядом с точкой максимума).

**`Box`**  
Прямоугольный блок с текстом — строительный элемент диаграмм. Координаты задаются в пиксельной системе (удобно для фиксированного layout диаграмм). Используется в `diagram`-сценах для представления понятий, определений, формул.

**`Arrow`**  
Стрелка между двумя объектами (Point или Box). Основной элемент для показа связей и переходов в диаграммах. Поле `label` позволяет подписать стрелку (например, "дифференцирование", "следует из").

**`SceneObject` (discriminated union)**  
```python
SceneObject = Annotated[
    Union[Point, Segment, Circle, Triangle, FunctionCurve, Label, Box, Arrow],
    Field(discriminator="type"),
]
```
Единый тип для поля `objects` в `Scene`. Pydantic выбирает нужный класс по значению поля `type` — это быстро и даёт понятные ошибки при неверном JSON.

### 3. `app/schemas/scene.py` — контракт pipeline

**Constraints (5 типов)**  
Constraints описывают дополнительные геометрические построения. LLM указывает смысл ("медиана из вершины A"), solver вычисляет координаты. Это ключевой принцип DSL: LLM не управляет координатами.

| Тип | Поля | Что строит solver |
|-----|------|-------------------|
| `MedianConstraint` | `triangle`, `vertex` | Отрезок из вершины в середину противоположной стороны |
| `AltitudeConstraint` | `triangle`, `vertex` | Перпендикуляр из вершины на противоположную сторону |
| `BisectorConstraint` | `triangle`, `vertex` | Биссектриса угла при вершине |
| `MidpointConstraint` | `segment`, `result_id` | Точка-середина отрезка |
| `RightAngleMarkerConstraint` | `vertex`, `ray1`, `ray2` | Маркер прямого угла (квадратик) |

**`InputRequest`**  
Входной запрос к сервису. `scene_type` обязателен и передаётся вызывающей системой — сервис не занимается классификацией типа визуализации (это ответственность внешнего сервиса, который знает контекст задачи).

**`Scene`**  
Основной контракт — структурированное описание того, что нужно нарисовать. Содержит холст, стиль, список объектов, список constraints и аннотации. Именно этот объект передаётся между компонентами pipeline: validator → solver → renderer.

**`RenderResult`**  
Ответ сервиса. `svg` — итоговая SVG-строка (или `null` при ошибке). `warnings` — нефатальные предупреждения (например, объект частично за пределами canvas). `metadata` — дополнительные данные для дебага и логирования.

### 4. `app/api/routes.py` + `app/main.py` — API

FastAPI-приложение с одним endpoint-заглушкой `POST /generate-svg`. Заглушка принимает `InputRequest` и возвращает `RenderResult` с `svg=None` и предупреждением `"not implemented"`. 

Зачем нужна уже сейчас: контракт API зафиксирован, можно интегрировать с вызывающей системой и писать тесты, не дожидаясь реализации рендера.

---

### 5. `tests/golden/` — 9 примеров JSON

По 3 примера на каждый тип сцены. Цель — зафиксировать ожидаемый формат входных данных до того, как появится LLM.

| Файл | Содержание | Что демонстрирует |
|------|------------|-------------------|
| `function_plot_1.json` | Парабола y = x² | FunctionCurve + Point |
| `function_plot_2.json` | y = sin(x) с подписью | Аннотации в annotations |
| `function_plot_3.json` | y = 2x + 1 с корнем | Линейная функция + точки пересечения |
| `geometry_1.json` | Треугольник + высота | Triangle + AltitudeConstraint |
| `geometry_2.json` | Окружность + медиана | Circle + MedianConstraint |
| `geometry_3.json` | Отрезок + середина | Segment + MidpointConstraint + Label |
| `diagram_1.json` | Схема производной | Box + Arrow с label |
| `diagram_2.json` | Свойства треугольника | Дерево блоков |
| `diagram_3.json` | Преобразования функций | Box + Arrow, три ветки |

В Milestone 1 эти файлы станут входными данными для golden-тестов: `POST /render-scene` должен возвращать валидный SVG для каждого из них.

---

### 6. `docs/scene-schema.md` — документация DSL

Полное описание всех типов объектов, таблицы полей с дефолтами, правила ссылок между объектами, типичные ошибки. Используется как:
- справочник для написания промптов (Milestone 3);
- эталон при написании validator (Milestone 1);
- документация API для внешнего сервиса.

### 7. `requirements.txt`

```
fastapi>=0.111.0
uvicorn[standard]>=0.30.0
pydantic>=2.7.0
```

Минимальный набор для запуска сервиса. Pydantic v2 выбран намеренно: discriminated union, `model_json_schema()` и `model_validate()` работают значительно быстрее v1, а structured output от LLM удобно валидировать через `model_json_schema()` напрямую.

## Что НЕ реализовано в Milestone 0

| Компонент | Когда |
|-----------|-------|
| Валидатор ссылок (проверка, что `from_point` существует) | Milestone 1 |
| Solver (вычисление координат constraints) | Milestone 1 |
| SVG renderer | Milestone 1–2 |
| LLM adapter и промпты | Milestone 3 |
| Golden-тесты как автотесты (`pytest`) | Milestone 1 |
